🔒 Confidentialité & RGPD

Données collectées

PulseUnit collecte uniquement les données nécessaires à son fonctionnement :

  • Compte — identifiant, prénom, nom, rôle professionnel, type d'agent (jour fixe / nuit fixe / alterné), code PIN haché (SHA-256, non réversible).
  • Garde — affectation aux lits, checklists, tâches IDE Tech, présence en ligne.
  • Planning personnel — états jour/nuit/CA/RH/RCV/HS/formation/maladie pour le calendrier individuel.
  • Bourse d'échange — annonces de permutation de gardes (date offerte/souhaitée, note libre).
  • Notes de lit — texte libre par lit, 5 emplacements par utilisateur, 7 jours.
  • Messages internes — conversations privées 1-à-1 + groupes par rôle (IDE/AS/Réa/USIP/Toute l'équipe). Texte, horodatage, accusés de lecture (✓ / ✓✓), réactions emoji, marqueur d'édition. Brouillons stockés uniquement en local (jamais transmis tant qu'ils ne sont pas envoyés).
  • Notifications — flux des alertes par utilisateur (titre, corps, type, lien d'action).
Stockage des données

Les données sont stockées dans Firebase Firestore (Google Cloud Platform, région europe-west, hébergement UE) et mises en cache dans le localStorage de votre navigateur pour le mode offline. Les notes de lit, le planning et les messages (DM + groupes) sont synchronisés entre vos appareils via Firestore. Les brouillons de messages et le cache de conversations restent en localStorage — non synchronisés. Aucun fichier audio, image ou pièce jointe n'est transmis.

Accès aux données

L'accès est restreint aux utilisateurs authentifiés via les Firestore Security Rules (request.auth != null, auth anonyme Firebase). Les notes de lit et les messages privés (DM) ne sont visibles que par leurs auteurs/destinataires. Les groupes par rôle sont visibles par tous les agents du rôle concerné (groupe « Toute l'équipe » accessible à tous). L'administrateur dispose d'un accès en lecture globale pour la supervision (gestion des comptes, déblocage, reset PIN). Aucune donnée n'est partagée avec des tiers ni revendue.

Notifications & Web Push

L'autorisation des notifications est demandée au moment de la création du compte (case à cocher décochable) et reste révocable à tout moment. Elle est gérée localement par votre navigateur (Notification API + Service Worker). PulseUnit n'envoie aucun token vers un serveur tiers (pas de Firebase Cloud Messaging dans la version actuelle). Les notifs sont déclenchées localement à partir des données Firestore que votre appareil reçoit.

Durée de conservation
  • Notes de lit — 7 jours (purge automatique côté client).
  • Notifications — 30 jours (purge automatique).
  • Réponses "travaillez-vous ?" — 4 jours.
  • Messages internes (DM + groupes) — persistants à vie tant qu'ils ne sont pas supprimés. Les DM peuvent être effacées par l'un des deux interlocuteurs (suppression visible des deux côtés). Les messages de groupe ne sont supprimables que par leur auteur.
  • Brouillons de messages — locaux, conservés tant qu'ils ne sont pas envoyés ou effacés. Vidés au logout local.
  • Données de garde, planning, compte — conservés le temps d'utilisation de l'application. Suppression sur demande via l'administrateur.
Sécurité

Codes PIN hachés SHA-256 côté navigateur avant envoi (jamais en clair). Connexion HTTPS. Content-Security-Policy en place. Compte bloqué après 3 codes erronés (déblocage admin). Service Worker en cache network-first pour index.html (mises à jour automatiques).

Vos droits (RGPD art. 13-14-15-16-17)

Vous disposez d'un droit d'accès, de rectification, d'effacement, de limitation, de portabilité et d'opposition. Pour exercer ces droits, contactez l'administrateur de l'application qui supprimera votre compte et toutes les données associées (notes, messages, planning, notifications) dans les meilleurs délais. Aucune décision automatisée ni profilage n'est effectué.

⚠️ Important — Données patient

PulseUnit est un outil d'organisation d'équipe. Ne saisissez JAMAIS d'informations patient nominatives (nom, prénom, date de naissance, NIP, diagnostic identifiant) dans les notes de lit, les messages internes (DM ou groupes) ou les annonces de bourse. L'application n'est pas un dossier patient et n'est pas conforme aux exigences HDS (Hébergeur de Données de Santé).
